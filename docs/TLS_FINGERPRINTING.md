# TLS/JA3 fingerprinting (backend: `rustls`)

Проект использует `rustls` для TLS. Параметр `tls.ja3_fingerprint` работает в режиме **best-effort** и **не** является полной имитацией TLS браузера (это не uTLS).

## На что движок реально влияет

Текущая реализация настраивает часть параметров рядом с ClientHello:

- порядок preference для cipher suites (через порядок crypto provider);
- порядок supported groups (key exchange);
- порядок ALPN (и частичная рандомизация для профиля `random`);
- предложение TLS 1.2/1.3 (в пределах ограничений конфига и ECH).

## Что нельзя полностью повторить (ограничения `rustls`)

Даже при тюнинге профиль может отличаться от реального браузера, потому что `rustls` контролирует:

- точный набор и порядок TLS-расширений (включая GREASE-поведение);
- порядок и состав signature algorithms;
- поведение KeyShare (состав/порядок);
- детали supported versions и legacy-полей;
- множество косвенных сигналов детекторов (кроме классического JA3): padding-паттерны, PSK-поведение и т.д.

## Профили `tls.ja3_fingerprint`

Поддерживаемые значения:

- `rustls_default`: без тюнинга;
- `chrome_120`: best-effort профиль, похожий на Chrome;
- `firefox_121`: best-effort профиль, похожий на Firefox;
- `random`: рандомизация части порядка в рамках ограничений `rustls`.

Если нужна максимально близкая имитация браузерного TLS, обычно требуется другой TLS-стек (например, клиент на базе uTLS). Для текущего `rustls`-backend это вне scope.
